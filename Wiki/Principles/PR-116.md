# PR-116 — Audit Result Trichotomy: every check must declare finding, clean, or could-not-verify

An audit that silently skips unverifiable checks is indistinguishable from a clean audit. The absence of a reported problem is not evidence of absence — it may be evidence of an incomplete run. Every check must produce exactly one of three explicit outcomes:

- **finding** — the check ran and identified a problem. Record exact locations and a plain-English statement of what it means in practice.
- **clean** — the check ran fully and found nothing. Recorded explicitly so coverage is visible.
- **could-not-verify (CNV)** — the check could not be completed (required file absent, tool unavailable, live resource not accessible, ambiguous evidence). Noted briefly with the reason. CNV ≠ clean.

Omitting the third state is the most common form of audit theater: a check that cannot run silently becomes a green result, giving false confidence.

Origin: vcohere/claude-code-marketplace `vibe-code-security-check` skill methodology (session 2026-06-23); previously the CC audit framework used only PASS/FAIL/WARN, lacking an explicit CNV state.

## Doctrine Snapshot

| Field | Value |
|---|---|
| **Symptom** | An audit framework has PASS and FAIL but no explicit CNV state; checks that cannot run silently appear as PASS |
| **Cause** | Binary thinking: the audit author assumes that "ran and found nothing" is the only alternative to "found a problem" |
| **Solution** | Add a third mandatory status (CNV) that surfaces when a check cannot be completed; treat CNV as neither PASS nor FAIL in the gate, but never hide it |
| **Action** | When implementing any audit check: define the condition under which it cannot be verified, and emit CNV with a reason string when that condition holds |
