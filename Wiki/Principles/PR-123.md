## Doctrine Snapshot

| Field | Value |
|---|---|
| **Symptom** | A verification or audit workflow (single-agent or multi-agent, parallel or sequential) confirms findings across multiple turns, but each confirmed finding is only summarized in chat/session output; registration into the canonical artifact (BACKLOG.md, an ADR, DECISIONS.md) is deferred until "everything is consolidated" at the end. |
| **Cause** | The intuitive urge to see the full picture before deciding structure (numbering, avoiding duplicate IDs, grouping related findings) is used to justify deferring registration itself, not just deferring the final organizational pass. |
| **Solution** | The registration cadence must match the confirmation cadence, not the workflow's overall completion. A confirmed finding is written into its canonical artifact in the same turn it is confirmed. If genuine uncertainty about final structure exists, a minimal entry is registered immediately, explicitly marked as pending consolidation — never left unregistered in chat. |
| **Action** | FORBIDDEN: summarizing a confirmed finding in chat and deferring its registration to a later turn "once all findings are in"; using a scratch/temp file as an intermediate draft for content that belongs in a canonical repo artifact (the same violation as using `/tmp` for decisions); treating "the finding is real and I said so" as equivalent to "the finding is registered". This is the same discipline PR-122 (Canonical Correction) applies to code fixes, applied instead to the *process* of closing an audit — holding a verified finding only in chat reproduces exactly the failure PR-122/S27 exist to prevent: if the session is interrupted mid-audit, unregistered findings are lost or exist only as unverifiable claims. |
