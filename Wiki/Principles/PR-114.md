# PR-114 — Equivalent-effect substitution: absence of the canonical path is a constraint, not a dead end

When the standard installation path for a tool or mechanism is unavailable (OS incompatibility, missing runtime, restricted environment), the correct response is to identify what effect the tool produces and achieve that effect through an available alternative — not to declare the feature impossible or skip enforcement entirely.

Origin: RTK hook installer unavailable on Windows native; CC's PreToolUse Bash matcher achieves identical token-compression enforcement natively. Session 2026-06-21.

## Doctrine Snapshot

| Field | Value |
|---|---|
| **Symptom** | Tool or hook cannot be installed via canonical path → enforcement silently dropped |
| **Cause** | Agent treats installation method as the only valid path to the effect |
| **Solution** | Identify the *effect* required, find an available mechanism that produces it, document the substitution explicitly |
| **Action** | Before declaring a feature impossible, enumerate alternative paths to the same effect. Document the substitution in AUDIT_TRAIL with the original constraint and the chosen equivalent. |
