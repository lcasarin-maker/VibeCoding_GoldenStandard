### Symptom (Signal)
> The system validates what comes IN (anti-junk-code, large-file, gates) but nothing hunts what must go OUT: refactors leave residue (dead imports/params, backups, phantom scripts, duplicate docs, orphan IDs in the catalog) that the gate approves just for existing

### Cause
Letter-of-the-law validation (`Path.exists`) instead of relevance; append-only culture; no retention policy or owner for the cleanup

### Application / Mitigation
Hunt orphans as a GATE (dead-code/vulture, allowlist without a file, dirs without retention, duplicate-purpose docs); REPLACE = `rm` + create in the same commit

### Operational Relevance
Govern output as strictly as input by hunting orphans and dead artifacts as a gate, not only blocking junk on the way in; the canonical guidance must stay published and linked from the index, which a CI link-check verifies.


### ❌ Example of the vice (Bad)
```bash
# The refactor leaves residue; the gate approves because the file 'exists'.
git mv old.py new.py        # dead old import, backup, and duplicate doc stay alive
```

### ✅ Corrected version (Good)
```bash
# Replace = delete + create in the SAME commit; hunt orphans as a gate.
vulture src/ && git rm old.py backups/old.py docs/old.md && git add new.py
```

### 🔎 Concrete detection
Output gate: dead-code (vulture), allowlists without a file, directories without
a retention policy, and duplicate-purpose docs; fails if a refactor leaves residue.
