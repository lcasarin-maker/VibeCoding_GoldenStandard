# TK-043: Entropía sin poda — gobernanza de entrada sin gobernanza de salida

| Campo | Detalle |
|---|---|
| **ID** | `TK-043` |
| **Categoría** | Tokenomics |
| **Estado** | **PREVENTED** |
| **Severidad** | **low** |
| **Profundidad** | 🟢 Deep |
| **Tags** | `tokenomics`, `prevented` |
| **Downstream Verification** | `none` |
| **Mecanismo de Validación** | `validate_golden_standard_catalogs.py --check-wiki` |

---

### Síntoma (Signal)
> El sistema valida lo que ENTRA (anti-código basura, large-file, gates) pero nada caza lo que debe SALIR: los refactors dejan residuo (imports/params muertos, backups, scripts fantasmaes, docs duplicados, IDs huérfanos en catálogo) que el gate aprueba por solo existir

### Causa (Cause)
Validación de letra (`Path.exists`) en vez de vigencia; cultura append-only; sin política de retención ni dueño de la limpieza

### Aplicación / Mitigación
Cazar huérfanos como COMPUERTA (dead-code/vulture, allowlist sin archivo, dirs sin retención, docs de propósito duplicado); REEMPLAZAR = `rm` + crear en el mismo commit

### Relevancia Operativa
Tokenomics cleanup validation now rejects orphan markdown files in Wiki/Vices and Wiki/Tokenomics so residual docs cannot linger unnoticed.

### ❌ Ejemplo del vicio (Bad)
```bash
# El refactor deja residuo; el gate aprueba porque el archivo 'existe'.
git mv old.py new.py        # old import muerto, backup y doc duplicado quedan vivos
```

### ✅ Versión corregida (Good)
```bash
# Reemplazar = borrar + crear en el MISMO commit; cazar huerfanos como compuerta.
vulture src/ && git rm old.py backups/old.py docs/old.md && git add new.py
```

### 🔎 Detección concreta
Compuerta de salida: dead-code (vulture), allowlists sin archivo, directorios sin
politica de retencion y docs de proposito duplicado; falla si un refactor deja residuo.

### 📚 Evidencia externa
- **vexp.dev / HumanLayer ACE** — el residuo no podado infla el contexto y el costo; la gobernanza de salida es tan necesaria como la de entrada.
