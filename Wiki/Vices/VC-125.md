### Symptom
> The session's learning is lost when context is closed or compacted, and the next session repeats already-resolved work.

### Cause
Treating the chat as durable memory instead of exporting decisions, findings, and pending items to a persistent ledger.

### Solution
Before closing a session with real work, export a structured retrospective with decisions, evidence, and pending items to a stable GS artifact.

### Corrective Action / Prevention
Require every new lesson to be externalized to the catalog or wiki before COMPACT/CLEAR; if there is no structured export, the closure remains incomplete.

### ❌ Example of the vice (Bad)
```python
# The session's learning lives only in the chat: lost on compaction.
# (no artifact) "we already understood the bug was the lock; the next session won't know"
```

### ✅ Corrected version (Good)
```python
# Before closing, export the retrospective to a stable ledger in the repo.
append_retro("Wiki/Project_Insights/", {
    "decision": "serialize access to lock X",
    "evidence": ".protocol/evidence/race.json",
    "todo": "add a concurrency test",
})
```

### 🔎 Concrete detection
Incomplete session closure if there was real work and no persistent artifact
(insight, catalog entry, or retro) with decisions and pending items was generated/updated.
