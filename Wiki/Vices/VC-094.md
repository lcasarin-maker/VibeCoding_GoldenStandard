### Symptom
> A config file (.env, .yaml, .yml, .json, .toml, .ini) that worked stops loading EXACTLY its first variable after being moved, re-saved, or copied. No error, no warning: the parser reads the first key with a leading BOM character glued to it, so it matches nothing and the loader silently falls back to an empty default.

### Cause
A UTF-8 BOM (EF BB BF) at the start of the file, invisible in any editor, introduced by Windows tools saving as "UTF-8" (Notepad, PowerShell 5.1's default Set-Content -Encoding UTF8). The system assumes the config file is what it appears to be; the failure surfaces far from its origin (an empty value at runtime, not a parse error).

### Solution
Reject a leading BOM in configuration files at write/commit time, and make critical loaders (secrets, credentials) fail loudly if an expected first key is missing after load -- defense in depth against this whole failure class.

### Corrective Action / Prevention
Static check on configuration files: the first 3 bytes must not be EF BB BF (UTF-8 BOM). When writing configs from PowerShell, use New-Object Text.UTF8Encoding($false) or Set-Content -Encoding utf8NoBOM (PS7), never the PS5.1 default "UTF8".

### ❌ Example of the vice (Bad)
```text
﻿AEQUITAS_SECRET_KEY=abc123
AEQUITAS_ALGORITHM=HS256
```

### ✅ Corrected version (Good)
```text
AEQUITAS_SECRET_KEY=abc123
AEQUITAS_ALGORITHM=HS256
```

### 🔎 Concrete detection
Read the first 3 bytes of the file; EF BB BF (decoded as a leading U+FEFF in the text form) means the file has a BOM. Applies to .env/.yaml/.yml/.json/.toml/.ini and any line-oriented config format.
