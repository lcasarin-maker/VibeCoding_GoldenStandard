### Symptom
> A line is excluded from a Semgrep rule via `# nosemgrep` or `# nosemgrep: rule-id` with no explanation of why the finding is safe to ignore at that specific call site.

### Cause
Semgrep flags a real pattern in code the author believes is safe (test fixture, trusted constant, already-validated input); suppressing is faster than adding a comment, and nothing enforces one.

### Solution
Every `# nosemgrep` suppression must carry a written justification on the same line explaining why THIS specific instance is safe, not just that the rule is noisy in general.

### Corrective Action / Prevention
Static source analysis flags a `# nosemgrep`/`# nosemgrep: rule-id` comment with nothing after it; the allowed pattern requires trailing justification text (e.g. `# nosemgrep: rule-id -- reason`).

### ❌ Example of the vice (Bad)
```python
x = eval(user_input)  # nosemgrep: python.lang.security.audit.eval-detected
```

### ✅ Corrected version (Good)
```python
x = eval(TRUSTED_CONFIG_LITERAL)  # nosemgrep: python.lang.security.audit.eval-detected -- TRUSTED_CONFIG_LITERAL is a hardcoded module constant, never user input
```

### 🔎 Concrete detection
Regex over source text matching `# nosemgrep` or `# nosemgrep: rule-id`, requiring non-whitespace text after the pragma on the same line.
