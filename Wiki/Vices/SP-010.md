### Symptom
> A knowledge repository (one whose product IS the YAML catalogs) accumulates protocol docs, generated outputs, and intake machinery at root, making it indistinguishable from project scaffolding and hard for agents to navigate.

### Cause
Canonical structure rules are written for project repos (src/, tests/, scripts/). Knowledge repos have a different root contract — the catalogs ARE the src — but this distinction is never stated, so each session decides ad-hoc what belongs at root.

### Solution
In a knowledge repo, root legitimately holds: YAML catalogs (the product), the primary generator script, and standard governance files (README, SPEC, HANDOFF, DECISIONS, STATE, .agents/, tasks/, audit/). Everything else is classified: protocol docs → knowledge/, generated JSON snapshots → output/, scripts → scripts/.

### Corrective Action / Prevention
Apply root audit to any knowledge repo using GS as a template. Any file at root that is not a YAML catalog, generator, or canonical governance file must move to its classified folder.

### 🔎 Concrete detection
audit.py (knowledge-repo mode): assert root contains only *.yaml catalogs, generator script, README/SPEC/HANDOFF/DECISIONS/STATE, and dot-folders. Flag any .md or .json file at root that is not in the allowed list.
