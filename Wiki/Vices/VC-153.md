### Symptom
> Operational sinks or cross-repo consumers are updated before the GS rule exists, forcing later backfills and making the sink behave like the source.

### Cause
The agent touches enforcement or generated outputs before the GS vice is authored and regenerated.

### Solution
Author the GS vice first, regenerate Wiki/report outputs, and only then update Cerberus consumers or enforcement hooks.

### Corrective Action / Prevention
Land the GS doctrine first; treat Cerberus references as downstream consumers that only follow after regeneration proves the rule exists.

### ❌ Example of the vice (Bad)
```yaml
# Cerberus references the rule id before the GS rule exists.
rule_id: VC-153
source: cerberus
```

### ✅ Corrected version (Good)
```yaml
# GS owns the rule first, then consumers cite it.
rule_id: VC-153
source: golden_standard
```

### 🔎 Concrete detection
Flags enforcement-layer edits that cite a missing or unregenerated GS rule.

### 📚 External evidence
- **BACKLOG.md cross-seam contract** — GS-2 must wait for a real GS gate, and cross-layer consumers should follow the GS source rather than create a sink-first shortcut.
