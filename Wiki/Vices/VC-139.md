### Síntoma (Symptom)
> La salida del modelo (SQL, HTML, comando, ruta o codigo) se ejecuta, renderiza o usa en un sink sin sanitizar, asumiendo que es segura.


### Causa (Cause)
Se trata la salida del LLM como confiable; no hay frontera entre 'texto generado' y 'accion ejecutada'.


### Solución (Solution)
Tratar la salida del LLM como NO confiable: parametrizar (SQL), escapar (HTML), allowlist (comandos/rutas) y validar contra esquema antes de ejecutar.


### Acción Correctiva / Prevención
Documentado; enforcement downstream. Es el lado OUTPUT de VC-074 (input). Mitigacion via taint analysis: semgrep / CodeQL.


### ❌ Ejemplo del vicio (Bad)
```python
sql = model.generate("query for user " + uid)   # salida del LLM
cursor.execute(sql)                              # ejecutada sin sanitizar -> inyeccion
```

### ✅ Versión corregida (Good)
```python
# Nunca ejecutar la salida cruda: derivar parametros validados y parametrizar.
filters = validate_schema(model.generate_json(...))
cursor.execute("SELECT * FROM t WHERE owner = ?", (filters['uid'],))
```

### 🔎 Detección concreta
Salida del modelo que fluye a un sink peligroso (cursor.execute con string, os.system, eval/exec, innerHTML) sin sanitizacion. Flow-dependent -> delega a semgrep/CodeQL (taint). Ver VC-074, VC-115, VC-127.
