### Symptom
> A coverage or traceability check reports a symbol as physically present when only a commented-out or string-literal mention of it exists; no real function or class backs it.

### Cause
The validator resolves a symbol by scanning raw source text (a substring or regex such as `"def <name>" in source`), which also matches inside comments and string literals. A planted `# def <name>` line games the check, deceiving the judge with text that no code backs.

### Solution
Resolve symbols structurally through the AST so that only a real FunctionDef or ClassDef node counts. A comment or string literal can no longer satisfy the validator.

### Corrective Action / Prevention
Resolve symbols structurally through the AST so only a real definition node counts; a commented-out or string mention must not satisfy the check.


### ❌ Example of the vice (Bad)
```python
# Validator: a symbol is "covered" if the text "def X" appears in the file.
# def audit_widget_safety   <- planted decoy; no real function exists
assert "def audit_widget_safety" in source  # passes on the comment alone
```

### ✅ Corrected version (Good)
```python
import ast
tree = ast.parse(source)
covered = any(
    isinstance(n, (ast.FunctionDef, ast.ClassDef)) and n.name == "audit_widget_safety"
    for n in ast.walk(tree)
)  # a comment or string literal can never satisfy this
```

### 🔎 Concrete detection
Coverage or traceability validators that confirm a symbol's existence with a
raw-text substring or regex over source (matching inside comments and string
literals), plus commented-out `def`/`class` lines that mirror a declared
validating mechanism.
