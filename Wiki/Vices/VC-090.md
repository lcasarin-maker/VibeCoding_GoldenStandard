### Symptom
> Interface, base class, factory, or layer with exactly one implementation; config option nobody sets; parameter accepted but ignored

### Cause
Agent anticipates future requirements that were never requested; "what if we need a second one later" reasoning without evidence

### Solution
No abstractions unless explicitly requested. Inline the single implementation. Add the abstraction when the second concrete case exists, not before.

### Corrective Action / Prevention
Before creating an interface, abstract class, factory, or plugin system: verify there are at least 2 concrete implementations today. One implementation = inline it. One config key with one value = hardcode it. One layer with one caller = delete the layer.


### ❌ Example of the vice (Bad)
```python
class AbstractRepository(ABC):        # yagni: one impl today
    @abstractmethod
    def find(self, id): ...

class PostgresRepository(AbstractRepository):
    def find(self, id): return db.get(id)
```

### ✅ Corrected version (Good)
```python
def find_user(id):
    return db.get(id)   # inline; add abstraction when second backend exists
```

### 🔎 Concrete detection
AST: abstract classes / Protocol types with exactly one non-abstract subclass in the repo. Interfaces exported from a module with a single concrete implementation. Config keys present in schema but set to a single value across all environments.
