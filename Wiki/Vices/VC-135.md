### Síntoma (Symptom)
> El modelo llama a un metodo, parametro o endpoint de una libreria que SI existe, pero que no existe en la version fijada (deprecado, renombrado o inventado por conocimiento desactualizado).


### Causa (Cause)
El conocimiento del modelo va desfasado respecto a la version instalada; nadie verifica la firma contra la API real.


### Solución (Solution)
Fijar la version y verificar cada simbolo contra la API real (docs up-to-date, MCP de documentacion, o import + inspeccion); rechazar llamadas que no resuelven.


### Acción Correctiva / Prevención
Documentado; enforcement downstream. Distinto de VC-129 (paquete inexistente): aqui el paquete existe pero el simbolo no. Mitigacion: docs up-to-date via context7 (PI-032).


### ❌ Ejemplo del vicio (Bad)
```python
# pandas 2.x elimino DataFrame.append; el modelo lo sigue usando.
df = df.append(row)            # AttributeError en la version instalada
```

### ✅ Versión corregida (Good)
```python
# Verificar contra la API real de la version fijada y usar el reemplazo vigente.
df = pd.concat([df, pd.DataFrame([row])], ignore_index=True)
```

### 🔎 Detección concreta
Resolver cada simbolo llamado (metodo/atributo/kwarg/endpoint) contra la API de la version instalada; un simbolo ausente = API alucinada u obsoleta. Sin detector self-contained (necesita la libreria); ver VC-129 y VC-056.

### 📚 Evidencia externa
- **context7 (upstash, verificado)** — existe para dar a los LLMs documentacion up-to-date precisamente porque generan llamadas a APIs obsoletas o inexistentes.
- **arxiv:2603.28592** — las APIs alucinadas figuran entre los patrones recurrentes del codigo AI-generado.
