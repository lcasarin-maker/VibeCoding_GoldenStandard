### Symptom
> Agent modifies files without a written step-by-step plan; steps are implicit in chat or agent memory, not in a durable artifact; no checkpoint to validate against and no signal when the task is complete.

### Cause
Optimism bias — agent trusts context memory over explicit checkpoints; planning feels like overhead when the task seems obvious; multi-step tasks started as single-step.

### Solution
Before any multi-file change, create PLAN.md with numbered steps and explicit acceptance criteria. Execute one step at a time. Delete PLAN.md in the closing commit — its absence is proof of completion. If PLAN.md survives the session, the session is not done.

### Corrective Action / Prevention
Before touching any file in a task with more than one change: (1) create PLAN.md with numbered steps and acceptance criteria; (2) execute one step at a time; (3) validate each step against PLAN.md before proceeding; (4) delete PLAN.md in the closing commit once all steps are done. Absence of PLAN.md after close = evidence of completion.

### ❌ Example of the vice (Bad)
```markdown
# Agent modifies 4 files and runs a migration with no PLAN.md.
# Steps exist only in chat. Context reset = no recovery path.
```

### ✅ Corrected version (Good)
```markdown
# PLAN.md before first file change:
# Step 1: Create .agents/AGENTS.md [acceptance: file parses, contains §0-§6]
# Step 2: Migrate BACKLOG.md [acceptance: BACKLOG.md deleted, tasks/backlog/ has N files]
# Step 3: Delete PLAN.md [acceptance: no PLAN.md in tree]
```

### 🔎 Concrete detection
Multi-file session with no PLAN.md at start. Secondary: PLAN.md present at session close (never cleaned up). Tertiary: plan exists but steps unnumbered or lack acceptance criteria.
