### Symptom
> A new package is installed to solve a problem already solvable by stdlib or an already-installed dependency

### Cause
Agent reaches for a familiar library without checking what is already available in the environment

### Solution
Before adding any dependency, climb the resolution ladder — stdlib first, then already-installed deps, then native platform features. Only if all fail, add a new dep.

### Corrective Action / Prevention
Resolution ladder before any new dependency: (1) does stdlib cover it? (2) does an already-installed package expose this? (3) does a native platform API cover it? Only after all three fail: add dep. Every new dep requires explicit justification in the PR why the ladder was exhausted.


### ❌ Example of the vice (Bad)
```python
# requirements.txt gains: python-dateutil
from dateutil.parser import parse
dt = parse("2026-06-26")   # stdlib datetime.fromisoformat exists since 3.7
```

### ✅ Corrected version (Good)
```python
from datetime import datetime
dt = datetime.fromisoformat("2026-06-26")   # stdlib, zero new deps
```

### 🔎 Concrete detection
Diff scan: any addition to requirements.txt / package.json / pyproject.toml. For each new dep, check if stdlib or existing deps expose equivalent functionality. native: tag — name the stdlib or installed replacement.
