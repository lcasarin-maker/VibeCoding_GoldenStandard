### Symptom
> The model generates code with insecure defaults: debug enabled, open CORS, TLS verification disabled, weak hashing, or binding to all interfaces without auth.


### Cause
The model optimizes for 'make it work' and emits the most permissive defaults; no one hardens the configuration before merging.


### Solution
Harden the defaults; review the generated code's security configuration against a checklist; block known insecure patterns in CI.


### Corrective Action / Prevention
Generated code must be secure by default: a static-signature detector, exercised against the catalog examples, flags insecure default patterns.


### ❌ Example of the vice (Bad)
```python
app.run(host="0.0.0.0", debug=True)        # debug in prod, bind to everything
requests.get(url, verify=False)            # TLS unverified
```

### ✅ Corrected version (Good)
```python
app.run(host="127.0.0.1", debug=False)
requests.get(url)                          # verify=True by default
```

### 🔎 Concrete detection
Static insecure-default patterns: debug=True, verify=False, CORS origins='*', hashlib.md5( for passwords, host='0.0.0.0'. Statically detectable (regex/AST); complement bandit/semgrep.
