### Symptom
> An erroneous or obsolete fact persists in the agent's durable memory (ledger, file, or memory MCP) and contaminates future sessions, or the memory becomes desynchronized from the real state.


### Cause
Persistent memory is written without verification, without a source, and without a date/version; it is not reconciled with the source of truth when loaded.


### Solution
Write to memory only verified facts, with source and verified_at; reconcile against the real state when loading; expire or invalidate the stale.


### Corrective Action / Prevention
Documented; enforcement downstream. Extends PR-042 (context poisoning) to DURABLE memory across sessions. Mitigation: memory layers like byterover / claude-context-memsearch (PR-111).


### ❌ Example of the vice (Bad)
```python
# A datum is persisted without verifying; 3 sessions later it still contaminates.
memory.write("endpoint = /v2/pay")    # never verified; it was /v1/pay
```

### ✅ Corrected version (Good)
```python
# Only verified facts, with source and date; reconciled when loading.
memory.write_fact("endpoint=/v1/pay", source="openapi.json", verified_at=today())
```

### 🔎 Concrete detection
Persistent memory entries without source/verified_at/version, or that contradict the current repo state when loaded. No self-contained detector (needs the store); see PR-042, PR-023, and PR-033.
